my_factorial <- function(n) {
  stopifnot( n >= 0, n == as.integer(n))
  if (n <= 1) return(1)
  return(n * my_factorial(n - 1))

debugonce() enters debugging once after the function is called and then never again.

debug(standardise) enters debugging everytime the function standardise is called. To not enter debugging after calling standardise you have to call undebug(standardise)

Scenario where debug() is prefered over debugonce():

When fixing a bug and planning on calling the function many times over.

You are debugging standardise() inside a larger analysis pipeline, and each small code change requires rerunning the function to see whether the problem is fixed.

Week 5 - Quarto Websites & Collaborative Coding

  1. What is the difference between a Quarto document and a Quarto website project? Describe in your own words what _quarto.yml does and how it changes how you think about organising a multi-file project.

A quarto document is a single page, where as a quarto website is a collection of different quarto documents.

_quarto.yml is the control settings of the whole Quarto website and structures everything. Instead of having to write the settings of the page at the top everytime, it is defined once in the _quarto.yml file. In my opinion, it is much more efficient and gives a structured look since the settings are the same across the website.

  1. Walk through one format: or execute: option you set in _quarto.yml this week. Why did you choose that option, and what effect did it have on the rendered output?

The format is html because I want the output to be a website.

code-fold: true - the code will not be directly shown. The user can click on the expandable to show the code. I chose this because it gives a more clean look instead of having pages of code.

code-summary: “Show code” - This message will appear on the expandable to specify that there is code and an option to show it.

code-line-numbers: true - It shows the line numbers inside of the code chunks. Also for structure.

code-copy: hover - When the user hovers over a code chunk it allows them to copy the code.

theme: - aesthetics of the website

  • cosmo - theme name

  • brand - allows custom changes on top of the cosmo theme

css: styles.css - a file that includes specific, custom style changes that are applied to every page

toc: true - Table of contents on the right side

  1. Describe a situation this week where you used a git branch, stash, or pull request. What problem were you solving, and would you approach it the same way again?

I was editing the main proposal file and was working on the main branch. Luckily only I was editing and pushing the changes so there were no conflicts but it could happen that the other teammate works on the branch and then it’ll lead to a merge conflict. To avoid this we decided to use individual branches when working on the project since it will lead to a cleaner history.

++ In week 8 I merged a feature branch into main and deleted the remote branch on GitHub. I continued working on that same branch locally and forgot that the remote tracking was gone. When I tried to push an error message appeared. I was very confused because I didn’t realize the remote branch no longer exists. In my attempt to fix the ‘sync’ issue, I improperly performed a pull that overwrote my local progress. I learned the importance of structured branches and frequent committing and pushing instead of doing the whole project on just one branch and pushing everything at once.

Next time we could resolve that problem like this:

  1. git branch shows I am on main instead of feature branch

  2. git log —oneline --graph --all -n 10 see if commits exist locally

  3. git fetch origin & git log --oneline origin/branchname..branchname check what was pushed –> If list of commits returns it means those commits exist locally and have not been pushed to the remote branch yet

  4. git push -u origin branchname –> push changes from the branch I’m actually on

  2. Did you ask an LLM about data licensing, provenance, or quality? What question did you ask, and how did you assess whether the answer was trustworthy?

  3. Describe one visualisation you made during IDA. What did you learn from it that a summary statistic alone wouldn’t have revealed?

    We made a visualisation of the unemployment trend over time. It showed us how the rate dropped or increased over the years. While a summary statistic includes the same numbers and one could draw the same conclusions the visualisation shows where the highs and lows were and the overall trend at one glance.

  4. What question did your data raise that you hadn’t anticipated at the start? Do you think it’s worth pursuing, and why?

    The data showed an abnormal high rate in the year 2005. The dataset alone couldn’t explain it and so we decided to research. It turned out to be this high because of a reform that counted the unemployed and people on welfare into one number starting in 2005. After this new rules were introduced where receiving welfare became stricter and so the numbers decreased again. We decided to not pursue this question or include it into one of our RQs since it can’t be explained by our data alone and we would have to use sources from outside.

Week 9 - Reproducibility, Open Data & renv

  1. Have you set up renv in a project (course or personal)? Describe one dependency or reproducibility issue it helped you avoid or document — or one challenge you encountered when getting it to work.

    Yes, we set up renv in our project. It helped us document our package versions in renv.lock, so the project can run the same way for everyone. I mostly used snapshot() after installing new packages. I didn’t use restore() consistently yet, so I sometimes installed missing packages manually instead. That’s something I want to fix going forward.

  2. Did you use a dataset with an explicit open licence (e.g. CC-BY, ODbL, PDDL)? What did the licence allow or restrict, and did that affect how you plan to share or cite it in your final project?

    Yes, we used a dataset under the Datenlizenz Deutschland Namensnennung 2.0 licence. It allows us to copy, use and redistribute the data, even commercially, as long as we attribute the source. This means we just need to cite Statistisches Amt München properly in our final project. This way it doesn’t affect our final project.

  3. Describe one documentation gap you discovered in your own or a team member’s analysis — something a future reader could not have reproduced or verified without asking you. How did you (or would you) fill it?

    One gap was in the column explenation. The raw file has columns like Basiswert 1, Basiswert 2 and Auspragung with no clear explanation of what they mean. A future reader couldn’t reproduce our filtering choices without the external metadata PDF, since it explains that Basiswert 2 changes meaning depending on Auspragung. We filled this gap by documenting it directly in our proposal, so readers don’t have to track down the PDF themselves.

Week 10 - Analysis & Inference

  1. Describe an analysis decision you made — model choice, variable inclusion, aggregation level, etc. What alternatives did you consider, and why did you choose what you did?

    For RQ1, we decided to group the 25 districts into five city areas instead of plotting all districts individually. We considered keeping all 25 districts separate, but that would need 25 plots and be hard to interpret. Grouping into five areas let us still see regional differences without overfitting to noise at the individual district level.

  2. If you used an LLM to help reason through a statistical question, describe the exchange. Did the LLM’s reasoning hold up? Where did it help, and where did you need to apply your own judgement?

    We used Gemini to help understand the infer function syntax, like specify() and assume(“t”). The first explanation was too complex, so we asked for a simpler version. That helped us understand what each function does, but we still had to apply our own judgement when debugging, since we had errors from using the wrong variable names. The LLM helped explain the syntax, not the actual statistical reasoning behind our result.

Week 11 — Statistical Communication & Visualisation

  1. What visualisation principle or technique did you apply this week that you hadn’t used before? Where did you learn it (lecture, reading, LLM, experimentation)?

    We applied the table versus plot decision this week. Before, we would default to plots for everything. Now we used a gt table specifically for the RQ2 comparison, because a table showed the exact means more clearly than a plot would. We learned this in the W11 lecture and practiced it first on the palmerpenguins sandbox in the practical.

  2. Describe a figure you revised or improved. What changed between the first and final version, and what drove each change?

    We revised our RQ plots by adding captions and interpretations to them. The first versions just showed the plot with a title and no further context. In the lecture and practical, we were taught to add captions and descriptions so a reader can understand the finding without having to interpret the plot themselves. This drove us to add captions and short interpretation sentences to our RQ1 and RQ2 plots, so each figure states what it actually shows instead of leaving that up to the reader.

  3. Think of a table or plot you made (or encountered) this week. Would the other format have communicated the same information better or worse — and why? If you used {gt}, what was one formatting choice you made deliberately (column labels, decimal places, spanners, source notes) and what guided it?

    For RQ2, a gt table communicated the difference in means better than a plot would, since the exact mean values are displayed. If we had used a plot instead, it would be harder to read off precise numbers. One deliberate gt choice was setting decimals to 1 for the rate columns, since more decimals would be false precision for a percentage like this.
