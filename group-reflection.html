Functions & refactoring (W2)

What we did:

We practices writing functions in Practical3.qmd. We wrote a grouped summary function using tidy evaluation (mean_mass_by() with ({{ }})-> searches for column name not the variable literally named) and a recursive function. We did not use custom functions into the group project itself. The actual analysis in proposal.qmd and report.qmd stayed as inline dplyr pipe chain code.

What worked well:

The practical exercise gave us a understanding of tidy evaluation ({{ }}) and recursion, which we hadn’t used before. Because it was separate from the project’s real analysis, we could focus on understanding without worrying about breaking anything that mattered.

What was hard:

Looking back, proposal.qmd has two places with duplicated code. The missingness check (count(Auspragung) and the NA summary) is repeated almost exactly. We run it once on the full dataset and again on the South Munich subset. The district code extraction (str_extract(Raumbezug, "^[0-9]+")) is also repeated. It appears in the RQ1 area mapping and again in the RQ2 South filter.

We didn’t notice this while writing the code. By the time we did, the deadline was close. Refactoring working analysis code just to make it tidier felt risky. We decided not to risk introducing a new bug just for style.

Debugging (W3)

What we did:

An example of a bug we encountered in the project is that rendered plot images were shown as broken/missing on the live GitHub Pages site. Our process through bugs was the same each time. We first read the actual error / looked at the problem and then checked the lecture/practical material for the topic involved (Quarto deployment for images) and attempted fixes ourselves. Only when that didn’t resolve it or made things worse, we asked an LLM how we could fix the problem and the cause.

For the broken images bug, trying to fix it ourselves didn’t get us far. The cause was a *_files/ rule in .gitignore silently excluding the rendered figure folders from every commit, so the HTML pages referenced image files that had never actually been pushed to the repo. The LLM correctly recognized this from the description alone, which we then confirmed by checking our actual .gitignore contents.

What worked well:

Trying to solve the problems ourselves first, using the actual error / problem and course materials, meant we had rough ideas what the fix / root could be for each bug before asking for help. This way we didn’t blindly trust the LLM and instead could make sense out of the explanation.

What was hard:

Most of the times the root causes weren’t obvious from what showed up alone. We wouldn’t have found the actual culprit by just looking at the code longer. The .gitignore was something we had set up at the very start of the project and forgot about since we didn’t fully understand what it does until actually deploying a website.

Version control & remotes (W4)

What we did:

We used Git for version control throughout the project, working with branches, commits, and a remote GitHub repository. Early on we hit a repo setup problem: the default branch was misspelled as “mian” instead of “main,” and the README had a stray placeholder word repeated several times. We tried fixing it with an LLM’s help, but the fix caused more problems than it solved, so we deleted the repository and started over. The same “mian” typo happened again on the second attempt, which led us to check GitHub’s repository creation settings directly rather than trying to fix it after the fact. We found the default branch name setting and corrected it there.

What worked well:

Once we adopted separate branches per task instead of everyone working directly on main, the workflow got noticeably cleaner and we avoided merge conflicts. Committing frequently also made it easier to structure work by finishing and committing one task before moving to the next kept the project from feeling chaotic.

What was hard:

Later in the project, we also ran into a remote tracking issue. After merging a feature branch into main and deleting the remote branch on GitHub, one of us kept working on that same branch locally without noticing the remote tracking was gone. A later push failed, and trying to resolve it, an improper pull overwrote local progress. We learned to check git branch and git status before continuing working.

Quarto websites & collaborative coding (W5)

What we did:

Everyone in the group worked on _quarto.yml. We didn’t split it up by person. Our workflow was to render the site locally first and check it looked right. Then we pushed it. GitHub Pages only updates from the rendered output, not the source files. So if we forgot to render before pushing, the live site would be out of date.

What was hard:

It took us a while to get used to the render-then-push workflow. We didn’t fully understand at first that GitHub Pages needs the rendered HTML, not just the source files pushed. This caused some early confusion.

Initial data analysis & data cleaning (W7)

What we did:

We used vis_dat, skim, miss_var_summary and vis_miss to screen the dataset. We found no unexpected column types, categorical variables were <chr> and numbers were <dbl>. We identified two data quality issues. First, individual districts and the Stadt München total were mixed in the same Raumbezug column. Second, the Auspragung column combined gender and totals (insgesamt) in one long column. We also found an external metadata PDF linked under the CSV file on the main data page, which explained that Basiswert 2 changes meaning depending on Auspragung, and that Stadt München rows aggregate differently than individual district rows.

What worked well:

The screening tools (skim, vis_dat, naniar) gave us a fast overview and made it easy to spot the missingness and structure issues early.

What was hard:

Getting familiar with the column names and what they actually meant took time, since names like Basiswert 1, Basiswert 2 and Auspragung gave no information on their own. It wasn’t hard once we got familiar and worked with the data some time.

Reproducibility, open data & renv (W9)

What we did:

We initialized renv in the project and committed renv.lock and the renv/ folder to the repo. When installing new packages, renv asked for a yes or no confirmation, and we followed that workflow with snapshot() to update the lockfile. For the dataset itself, we used data under the Datenlizenz Deutschland Namensnennung 2.0 license, which allows use and redistribution as long as the source is attributed. The license is public, so we didn’t run into any issues.

What worked well:

Committing the lockfile itself was simple and snapshot() picked up our packages without issues.

What was hard:

The renv syntax was confusing at first, especially the yes or no prompts when installing new packages. We also didn’t consistently use renv::restore() after pulling changes from each other, so we didn’t fully close the loop on reproducibility. We mostly relied on installing packages manually if something was missing, instead of restoring from the lockfile. We intend on using restore() from now on.

Analysis & inference strategies (W10)

What we did:

For RQ1, we mapped each district to one of five city areas (Center, North, East, South, West) using the district code, joined that onto the dataset, and calculated annual mean unemployment rates per area from 2000 to 2024. We added Stadt München back in separately as a “Total” line. We produced a faceted line plot, plus an interactive plotly version with hover tooltips showing the exact year and rate.

For RQ2, we ran a difference in means test comparing männlich and weiblich unemployment rates in Munich for 2024, using infer with a simulation based permutation null distribution (5000 reps), shade_p_value(), and a significance level of \(\alpha = 0.05\). We also built a summary gt table.

We originally considered a third question, whether we could build a simple model to predict future unemployment percentages. We cut this to keep the project focused on two questions, in line with the course guidance to limit the proposal to at most two RQs.

What worked well:

The area mapping and annual means calculation for RQ1 worked cleanly. RQ2 results gave us a concrete p-value for the Hypothesis decision and a plot for visualization.

What was hard:

For RQ2, we changed our initial plan from just looking at South Munich to looking at all 25 districts. We made this decision because when we ran the Hypotheses test, the result was strongly against the Null Hypothesis, that there is no systematic disparity between women and men in unemployment. We became a bit skeptic because the sample size was also very small and so we ran the test on Munich as a whole. The result was very different with no evidence that there is this systematic disparity. We decided on keeping the second one since it was more reliable because of the sample size (25 instead of 6 districts). Additionally, we used a simulation based permutation test instead of a t-test, because rates / probabilities are often skewed and don’t follow a normal- or t-distribution.

We used Gemini to help understand the infer function syntax because it was hard to understand just from the practical. The first explanation was too complex, so we asked for a simpler version. That helped us understand what each function does, but we still had to apply our own judgement when debugging, since we had errors from using the wrong variable names and our functions were not complete. The LLM helped explain the syntax, not the actual statistical reasoning behind our result.

For the {infer} test, we used Google Gemini to discuss whether a theory-based t-distribution was appropriate given we have percentages. Based on this discussion, we rethought using the t-test versus a simulation based permutation approach on https://moderndive.com/v2/hypothesis-testing.html and decided to implement the simulation based test.

Key analytical decisions

Decision 1: Expanding RQ2 to all 25 Munich districts

What were we deciding between?

We were deciding whether to test the gender gap in unemployment across all of Munich, or narrow it to one specific area (South) and what {infer} test to use.

How did we reason through it?

We decided on Munich as a whole, since focusing on just South Munich’s districts would limit the sample size to just 6 districts. This could bear the risk of including or running into distorted noise data. To have a more stable test and output we decided on the biggest possible sample.

For the test we discussed with Google Gemini and verifyed the recommendation of a simulation based permutation test by looking through ModernDive’s hypothesis testing chapter .

In hindsight, would we decide differently?

No, we’re happy with the scope, since it gave us a clear, focused result instead of a by chance distorted mislead and the test delivered results without assumptions of a distribution.

Decision 2: Grouping districts into five city areas for RQ1

What were we deciding between?

We were deciding between plotting all 25 districts individually or grouping them into fewer, larger regions.

How did we reason through it?

25 individual plots would be hard to interpret at a glance. We built our own mapping of districts into five city areas (Center, North, East, South, West) so we could still see regional differences without having to look at 25 individual plots. We also kept the Stadt München total as its own separate “Total” category for completion.

In hindsight, would we decide differently?

No, this grouping is a good balance of not too much detail and not too little.

Decision 3: Cutting RQ3

What were we deciding between?

We were deciding between keeping a third research question on predicting future unemployment percentages or limiting the project to just two RQs.

How did we reason through it?

We decided to cut it to keep the project focused on two questions, in line with the course guidance to limit the proposal to at most two RQs. We also noticed that a prediction would be risky, because events outside the data, like economic shocks, could have a major impact on the trend and can’t be captured by a simple model.

In hindsight, would we decide differently?

No, cutting it was the right call, since it let us go deeper on RQ1 and RQ2 instead of spreading effort across three questions.

Workflow & collaboration

How did we divide work across group members?

We split tasks within each research question rather than assigning whole RQs to one person. For example, for RQ2, Rebecca built the gt table while Anna programmed the infer test. This let us work in parallel on the same question instead of one person owning it end to end.

What branching or Git workflow did we actually use — and did it match the proposal?

We mostly followed the plan from the proposal, using separate branches per task (e.g. feature/RQ1) and having another teammate verify before merging. This didn’t always hold up though. When something needed to be fixed fast or done quickly, we pushed directly to main without going through review. So the branching workflow was consistent for planned work but broke down under time pressure.

What would we change about how we coordinated?

We would structure task assignments more clearly. Right now we mostly divided work informally, saying things like “I’ll do this, you do that, could you look into that,” instead of writing tasks down with clear ownership and scope. More structure here would reduce the chance of things falling through the cracks or people working on overlapping parts by accident.

LLM usage

Describe up to 5 significant uses of an LLM tool during the project.

Usage 1: Debugging GitHub Pages not updating

  • Tool / model: Google Gemini
  • What it produced: A list of possible reasons why the site wasn’t updating. After we gave more context (that everything was already pushed to the repository), it narrowed this down further.
  • What we had to correct or verify: The first response was too generic since it didn’t account for the fact that we had already pushed. We had to give it our specific situation before the explanation became useful and we verified the actual fix ourselves by rendering and then pushing.

Usage 2: infer function syntax and programming support

  • Tool / model: Google Gemini
  • What it produced: A structured explanation of what each function in the infer practical template does (specify(), generate(), etc.). After we asked for a simpler explanation, it gave a clearer version. A draft of the infer code for the simulation test, based on our decision to switch from the theory-based t-test.
  • What we had to correct or verify: The first explanation was too complex to follow. After getting a simpler version and implementing it ourselves, we ran into errors from using the wrong variable names and using the functions wrong, which we then debugged with its help. We verified the final code produced the correct output by checking it against our actual research question. We also had to check the reps = 5000 and set.seed(42) were doing what we thought (shuffling 5000 times and keeping the shuffling the same each time the code is run) and made sure the filtering matched our RQ before trusting the output.

Usage 3: Grammar and phrasing in written sections

  • Tool / model: Claude
  • What it produced: Suggestions for fixing awkward sentence structure and word choice in written sections, while keeping our own meaning.
  • What we had to correct or verify: We explicitly asked it to keep our meaning and only fix grammar and word choice, since we sometimes used words that weren’t precise enough. We reviewed each suggestion and only kept changes that matched what we actually meant. Sometimes we changed word suggestions we felt unnatural with.
